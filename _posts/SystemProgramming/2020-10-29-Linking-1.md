# 본격적인 시작에 앞서
이어질 논의들에 앞서 링킹을 설명하는 동안 쭉 예시로 사용할 짤막한 코드를 소개하고자 한다.

```c
// main.c
int sum(int* a, int n);

int array[2] = {1, 2};

int main() {
    int val = sum(array, 2);
    return val;
}
```

```c
// sum.c
int sum(int* a, int n) {
    int i, s = 0;

    for (i = 0; i < n; i++) {
        s += a[i];
    }

    return s;
}
```

당연한 이야기지만, 링킹은 여러 소스 파일들을 한데 묶어 하나의 `실행 파일(executable)`을 생성하기 위한 과정이므로 예시는 `main.c`와 `sum.c`, 두 개의 C 소스 파일로 구성되어 있다.
코드에 대해 간략히 설명하자면, `main.c`에서는 `sum`이라는 함수와, 전역변수 `array`가 선언되어 있으며 `main` 함수 내에선 `sum`이라는 함수를 호출한 후 그 결과인 `val`을 반환한다. 


여기서 주의깊게 보아야 하는 것은
`main.c`에는 함수 `sum`의 선언(어떤 타입의 인자들을 받고, 함수의 이름은 무엇이며, 어떤 타입의 값을 반환하는지)만 존재한다는 것이다. 이 파일 어느 곳에도 `sum`이라는 함수의 정의가 없다. 그럼에도 `main` 함수 내에서
함수 `sum`을 당당하게 호출하는 이유는, 이 코드를 짠 프로그래머가 어딘가 믿는 구석이 있다는 것이다. 


그렇다! **프로그래머는 `main.c`와 `sum.c`라는 파일이 잘 연결(link)되어, `main` 함수 내의 `sum`이 `sum.c`에 명시된 대로 작동할 것을 기대하고 있다.** 그리고 링커는 이 프로그래머의 기대에 부응하기 위해, 저 두 코드를 잘 연결해 하나의 실행 가능한 바이너리를 만들어낼 것이다. 


그럼, 링킹이 구체적으로 어떻게 되는지 알아보기 전에 실행 가능한 바이너리를 만드는데 어떤 프로그램이 어떤 순서로 관여하는지 살펴보자. 우리가 짠 소스 코드를 기계가 읽을 수 있게끔 번역하는 과정에는 네 번역가 및 편집자들이 참여한다. 순서대로 `전처리기(preprocessor)`, `컴파일러(compiler)`, `어셈블러(assembler)`, 그리고 `링커(linker)`다.

# 컴파일러 드라이버
대부분의 컴파일 시스템(compilation system)은 사용자가 전처리기, 컴파일러, 어셈블러, 링커를 호출할 수 있게끔 `컴파일러 드라이버(compiler driver)`를 제공한다. 예를 들어, GNU 컴파일 시스템에서 앞서 설명한 두 코드를 실행 가능한 프로그램으로 빌드하기 위해선 `GCC(GNU Compiler Collection)`라는 컴파일러 드라이버를 사용하게 된다. 이를 사용하기 위해선 간단히 리눅스 터미널에


`$ gcc -Og -o prog main.c sum.c`


를 입력하면 된다. 그러면 아래 나열된 과정대로 ASCII 소스 파일(= 사람이 읽기 쉬운 소스 코드)이 여러 중간 형태를 거쳐 컴퓨터가 읽고 실행할 수 있는 바이너리로 번역된다. 여기서는 C언어를 기준으로 설명한다.

1. C 전처리기(C Pre-Processor, cpp)가 소스 파일에 있는 매크로를 처리해 .i 확장자의 중간 파일로 변환한다.

2. C 컴파일러가 C언어로 작성된 코드를 어셈블리어로 번역한다. 

3. 2에서 완성된 어셈블리 코드를 어셈블러가 기계어로 번역해 `목적 파일(object file)`을 만든다.

4. 빌드 과정에 필요한 모든 C 소스 파일에 대해 2와 3을 반복한다.

5. 모든 C 소스 파일로부터 목적 파일이 만들어졌다면, 링커가 이들을 링킹하여 실행 파일을 만든다.

앞에서 살펴본 명령어 한 줄만으로 이 다섯 단계를 한꺼번에 실행할 수도 있지만, 아래의 명령어들을 이용해 각 단계를 따로 따로 진행하는 것도 가능하다. 

1번 단계의 경우:

`$ cpp [other arguments] main.c /tmp/main.i`  
와 같이 cpp만을 단독으로 호출해 `main.c`라는 파일을 전처리하여 tmp 디렉토리 아래에 `main.i`라는 파일로 저장하게 만들 수 있다. 그 다음으로 호출해야 하는 것은 C 컴파일러로서:

`$ cc1 /tmp/main.i -Og [other arguments] -o /tmp/main.s`  
이렇게 되면 이전에 생성한 `main.i`를 C 컴파일러가 어셈블리어로 번역해서 그 결과물인 어셈블리어 소스 파일(확장자 `.s`)을 tmp 폴더 아래 디렉토리에 저장하게 된다. 이제는 어셈블리어를 최종적으로 기계어로 번역할 차례다:

`$ as [other arguments] -o /tmp/main.o /tmp/main.s`  
이로써 tmp 디렉토리 아래에 `main.o`라는, `main.c`를 기계어로 번역한 목적 파일이 생성되었다. 이와 같은 과정을 `sum.c`에 대해서도 거치게 된다.

지금까지 `main.c`와 `sum.c`를 각각 기계어로 번역했으므로, 각 파일은 서로에 대해서 모르는 상황이다. **당연하게도 `main.o`와 `sum.o`는 개별적으로 실행할 수 없다. 아직까지도 이들은 실행 파일을 만들기 위한 중간 산물에 불과하다.**


생각해보면 이는 당연한 일인데, 아직 `main.o` 내의 명령어들 중 불확실한 부분들이 남아있기 때문이다. 소스 코드에서 볼 수 있듯 `main.c`에는 함수 `sum`의 정의가 없었다. 기계어로 번역된 지금도, `main.o`라는 파일은 `sum` 함수를 구성하는 명령어가 어딨는지 모른다. (물론 우리는 이 명령어들이 `sum.o`에 있다는 걸 안다.) 따라서 `main.o`와 `sum.o`는 서로 연결되어야 한다. 그래야만 `main.o`에서의 함수 `sum`에 대한 참조(reference)가 정상적으로 이루어질 수 있기 때문이다. 


따라서 우리는 마지막으로 링커를 호출해 두 파일을 연결하고, 실행 파일을 만들게끔 한다.

`$ ld -o prog [system object files and args] /tmp/main.o /tmp/sum.o`  
이로써 우리는 `prog`라는 이름의 실행 파일을 만들었고, 잘 돌아가는지 확인하기 위해 리눅스 터미널에 다음과 같이 입력해보면:  

`$ ./prog`  
프로그램이 정상적으로 실행되는 것을 볼 수 있다. 물론, `prog`라는 프로그램을 실행하게끔 하는 과정에도 복잡한 메커니즘이 관여하지만, 이 부분에 대해선 뒤에 `로더(loader)`를 설명하며 다루겠다.
